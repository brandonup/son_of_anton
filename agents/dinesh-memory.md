Implementation lessons and gotchas. Updated at session end.

- [2026-03-30] [kinetic] **For Deno Edge Functions, implement MCP JSON-RPC 2.0 directly rather than importing the MCP SDK.** The official `@modelcontextprotocol/sdk` has Node.js transport assumptions (stdio, express) that don't map to Deno's `Deno.serve()`. Direct JSON-RPC dispatch is ~60 lines and avoids compatibility issues.
- [2026-03-30] [kinetic] **Python apps with `unstructured[all-docs]` need a Dockerfile on Railway — nixpacks fails.** The package requires poppler, tesseract, libreoffice, pandoc as system deps. Set `builder = "DOCKERFILE"` in `railway.toml` and install native deps in the Docker build stage.
- [2026-03-29] [kinetic] **When a column is added to `000_complete_schema.sql`, also create a standalone incremental migration file.** `000_complete_schema.sql` is for fresh DB setup only — existing databases need a separate `.sql` file (e.g., `20260329000007_add_debug_prompt_to_messages.sql`) to pick up the change via the Supabase SQL Editor. Without it, the column exists in the schema spec but not in production.
- [2026-03-29] [kinetic] **Admin observability features need both a write path and a read path — verify both exist.** `debug_prompt` had a migration, a read endpoint (`admin_prompt_debug.py`), and tests — but the generation engine never wrote the value. Always trace: where does this field get written? If the answer is "nowhere yet", the feature is incomplete regardless of schema and endpoint coverage.

- [2026-03-28] [kinetic] **Supabase `.single()` throws `PGRST116` (500) when 0 rows match — use `.maybe_single()` for ownership checks.** `_verify_company_ownership` in conversations.py used `.single()` which raised an unhandled `APIError` instead of returning `None`. Any "does this row exist?" query that may return 0 rows must use `.maybe_single()`.
- [2026-03-28] [kinetic] **`agent_definitions.owner_id` is a user UUID, not a company_id.** The agent chat page passed `agent.owner_id` as `company_id` when creating conversations, causing silent 403s. Agent definitions don't carry a company_id — fetch the user's companies separately.
- [2026-03-28] [kinetic] **The `generate` endpoint stores both user and assistant messages — don't store the user message separately before calling it.** `POST /api/v1/conversations/{id}/generate` handles user message insert (step 5), context assembly, LLM streaming, and assistant message insert. Calling `store_message` first duplicates the user message.
- [2026-03-28] [kinetic] **New DB columns referenced in code but not yet migrated cause `PGRST204` on insert.** `debug_prompt` on `messages` table was referenced in generation.py but the migration wasn't applied, breaking assistant message storage. Always verify column existence in the live DB before referencing new columns in insert/update operations.
- [2026-03-28] [kinetic] **`admin_models.py` has a module-level cache (`_models_cache`) — seeding the DB after server startup leaves the cache stale.** If `llm_models` (or any other cached table) appears empty in the UI after a migration, restart the API server to clear the cache. The cache resets to `None` on restart and re-queries on next GET.
- [2026-03-28] [kinetic] **Migrations in `packages/api/migrations/` are applied manually via the Supabase SQL Editor, not via CLI.** The `supabase/migrations/` directory is only used for RPC-style migrations. New `.sql` files in `packages/api/migrations/` must be copy-pasted and run by Brandon.
- [2026-03-28] [kinetic] **Supabase RLS blocks browser-side queries on `public.users` — use `SECURITY DEFINER` RPCs instead.** The anon-key client can't read the `users` table even for the authenticated user's own row (500 from PostgREST). Created `is_admin()` RPC with `SECURITY DEFINER` to bypass. Any future browser-side role/permission checks should use RPCs, not direct table queries.
- [2026-03-28] [kinetic] **When committing 200+ files spanning many sessions, exclude `.claude/launch.json` (local IDE config).** Use `git add --all && git reset HEAD .claude/launch.json` since `git add --all -- ':!path'` pathspec exclusion doesn't work reliably.
