# Son of Anton — Conventions

Last updated: 2026-03-29

These apply across all Son of Anton projects.

---

## Code

### Language Defaults

- **Python** for backend services, scripts, data pipelines, and GenAI application logic
- **TypeScript** (strict mode always on) for frontend and any Node.js services
- **React** for UI (functional components, hooks — no class components)
- No JavaScript without TypeScript. No exceptions.

### Python Style

Python 3.11+. `ruff` for linting/formatting. Type hints on all signatures. Pydantic for validation. No bare `except:`. Google-style docstrings.

### TypeScript Style

`strict: true` always. No `any` without comment. `const` over `let`, never `var`. Named exports over default. When `npx` is sandbox-blocked, use `./node_modules/.bin/<tool>` directly.

### Testing

- Every feature ships with tests
- Python: `pytest` with `pytest-asyncio` for async code
- TypeScript: `vitest` preferred; `jest` acceptable
- Coverage target: 80% minimum; 100% for core business logic
- Test file naming: `test_[module].py` or `[module].test.ts`

### API Design

- REST by default
- Versioning: `/api/v1/...`
- Error shape: `{ error: { code, message, details? } }`
- Validate inputs at the API boundary (Pydantic for Python, Zod for TypeScript)

## Git

### Commit Messages

Format: `[type]: [imperative description]`

Types: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`

One commit per logical change. Don't batch unrelated changes. Don't commit commented-out code.

### Commit Instructions to Brandon

When telling Brandon to run a commit, always provide the full shell command — not just the filename. Brandon copy-pastes from the terminal; file paths alone require him to look up the command.

**Correct:** `git add packages/api/routes/agents.py && git commit -m "feat: add agent activation endpoint"`
**Incorrect:** "Add and commit the agents file."

### Branches

- `main` is always deployable
- Feature and fix branches: use the `gitBranchName` auto-generated by Linear
- No direct commits to main
- **Sandbox blocks git index operations.** When blocked, write a shell script to `/private/tmp/claude-501/` and tell Brandon to run it immediately.

### PRs / Code Review

- PRs require review before merge
- PR description: what changed, why, how to test locally

## Documents

### File Naming

- Lowercase, hyphen-separated: `prd-decision-review.md`, `adr-001-database-choice.md`
- ADRs numbered sequentially: `adr-001-`, `adr-002-`, etc.
- PRDs named by feature: `prd-[feature-name].md`
- Reviews date-prefixed: `2026-03-09-auth-review.md`

### Document Status

All PRDs and ADRs must have a status header: `Draft | In Review | Approved | Deprecated`

### Markdown

H2 for top-level, H3 for subsections. Tables for comparisons. No padding paragraphs.

## GenAI-Specific

### Prompt Management

- Prompts are code. Version-control them.
- No hardcoded prompts in application logic — use a prompts module
- Every prompt gets an ID and version: `decision-review-v1`, `signal-extract-v2`

### Model Selection

- Default to the smallest model that handles the task reliably
- Document model choices and rationale in the relevant ADR

### Evals

- Every LLM-powered feature needs at least a basic eval before shipping
- Store evals in `projects/[project-name]/evals/`

## MEMORY.md Entry Rules

Keep MEMORY.md small. Bloated memory files degrade agent performance.

- **Max 2 sentences per entry.** If you need more, write a doc and reference it.
- **Never duplicate spec content.** Format: `[Topic] decided. See [doc].`
- **Delete superseded entries** outright.
- **At session end**, compress any entry you wrote that exceeds 2 sentences.
