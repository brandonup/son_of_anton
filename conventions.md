# Son of Anton — Conventions

Last updated: 2026-03-09

These apply across all Son of Anton projects. Agents read this file when producing code, docs, or other outputs. When a project diverges from these conventions, note it in that project's MEMORY.md.

---

## Code

### Language Defaults

- **Python** for backend services, scripts, data pipelines, and GenAI application logic
- **TypeScript** (strict mode always on) for frontend and any Node.js services
- **React** for UI (functional components, hooks — no class components)
- No JavaScript without TypeScript. No exceptions for "quick scripts" that end up in the codebase.

### Python Style

Python 3.11+. `ruff` for linting/formatting. Type hints on all signatures. Pydantic for validation. No bare `except:`. Google-style docstrings.

### TypeScript Style

`strict: true` always. No `any` without comment. `const` over `let`, never `var`. Named exports over default. When `npx` is sandbox-blocked, use `./node_modules/.bin/<tool>` directly (e.g., `./node_modules/.bin/vitest run`).

### Testing

- Every feature ships with tests. No exceptions.
- Python: `pytest` with `pytest-asyncio` for async code
- TypeScript: `vitest` preferred; `jest` acceptable
- Coverage target: 80% minimum; 100% for core business logic
- Test file naming: `test_[module].py` or `[module].test.ts`
- Unit tests for logic; integration tests for API endpoints; skip e2e unless explicitly planned
- Duplicate class names in the same pytest file silently shadow the first class — all its tests go unrun with no warning. Always use unique class names per file.

### Error Handling

- Surface errors explicitly — don't swallow them silently
- **Never `return None/[]/False` in a `try/except` on write operations — raise or log-and-raise instead.** Silent swallowing on state-changing operations (DB writes, API mutations, file operations) masks data loss. Read-path fail-open (return `[]` on a failed search) is acceptable when documented with a comment.
- Use structured logging (JSON format in production)
- Every API endpoint needs explicit error handling with meaningful status codes and messages
- Dinesh / Big Head: if you're tempted to skip error handling to ship faster, flag it as technical debt in MEMORY.md instead

### API Design

- REST by default; GraphQL only if multiple clients need flexible querying
- Versioning in the URL path: `/api/v1/...`
- Always return consistent error shapes: `{ error: { code, message, details? } }`
- Validate inputs at the API boundary (Pydantic for Python, Zod for TypeScript)

### Supabase in Async Code

All Supabase client calls inside `async def` methods must use `run_in_executor`:
```python
await asyncio.get_running_loop().run_in_executor(None, lambda: client.table(...).execute())
```
Never call the sync client directly in async context. Use `get_running_loop()`, not `get_event_loop()` (deprecated in 3.10+).

### Python↔TypeScript Field Mapping

Python uses snake_case (`workspace_id`). TypeScript uses camelCase (`workspaceId`). No automatic serialization layer exists. Every field crossing the BFF boundary must be manually mapped. Watch for this in SSE events (`session_id` vs `sessionId`, `message_id` vs `messageId`).

### Canonical Schema Validation

Before submitting any code that creates or references tables, verify all table names, column names, and constraints against `docs/db-schema-spec.md`. Inline DDL in specs may diverge from the canonical schema — always use the schema spec as source of truth.

## Git

### Commit Messages

Format: `[type]: [imperative description]`

Types: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`

Examples:
- `feat: add decision review engine API endpoint`
- `fix: handle null response from intelligence layer`
- `refactor: extract signal scoring into separate module`
- `docs: update MEMORY.md with auth decision`

One commit per logical change. Don't batch unrelated changes. Don't commit commented-out code.

### Branches

- `main` is always deployable
- Feature and fix branches: use the `gitBranchName` auto-generated by Linear (visible in `get_issue` response). This auto-links PRs to issues.
- No direct commits to main
- **Sandbox blocks git index operations.** `git add`, `git commit`, and `git mv` fail with "Unable to create .git/index.lock: Operation not permitted". When blocked, write a shell script to `/private/tmp/claude-501/` and tell Brandon to run it immediately — do not defer.

### PRs / Code Review

- PRs require Gilfoyle review before merge (even if it's just a quick look)
- PR description must include: what changed, why, how to test locally
- No "fix review comments" commits — squash or amend before merge

## Documents

### File Naming

- Lowercase, hyphen-separated: `prd-decision-review.md`, `adr-001-database-choice.md`
- ADRs are numbered sequentially: `adr-001-`, `adr-002-`, etc.
- PRDs are named by feature: `prd-[feature-name].md`
- Reviews: date-prefixed: `2026-03-09-auth-review.md`

### Document Status

All PRDs and ADRs must have a status header: `Draft | In Review | Approved | Deprecated`

No document moves to implementation without status = `Approved`.

### Markdown

H2 for top-level, H3 for subsections. Tables for comparisons (not bullet lists). No padding paragraphs. Plain language — define jargon on first use.

## GenAI-Specific

### Prompt Management

- Prompts are code. Version-control them.
- No hardcoded prompts in application logic — use a prompts module or config file
- Every prompt gets an ID and version: `decision-review-v1`, `signal-extract-v2`
- When changing a prompt, document what changed and why in MEMORY.md

### Model Selection

- Default to the smallest model that handles the task reliably
- Document model choices and rationale in the relevant ADR
- If a task requires a larger model, flag the cost implication

### Evals

- Every LLM-powered feature needs at least a basic eval before shipping
- Start simple: a set of test inputs with expected output characteristics
- Store evals in `projects/[project-name]/evals/` when a project accumulates them

## Project Management

### Linear as Source of Truth

All Son of Anton projects are tracked in Linear. Each project maps to a Linear team (e.g., Kinetic → **Kinetic** team). Linear is the authoritative record of what's being worked on, what's blocked, and what's done. MEMORY.md remains the authoritative record of decisions and context.

### MEMORY.md Entry Rules

Keep MEMORY.md small. Bloated memory files degrade agent performance — context used on history is context lost for actual work.

- **Max 2 sentences per entry.** If you need more, the detail belongs in a spec — write the entry as a 1-line reference to that doc.
- **Never duplicate spec content.** Format: `[Topic] decided. See [doc].`
- **Delete superseded entries** outright — no strikethroughs, no archives.
- **When writing a new entry that contradicts an existing one**, delete the old one in the same edit.
- **At session end**, compress any entry you wrote that exceeds 2 sentences.

### Labels

Every project uses these five agent labels, in addition to the standard Bug / Feature / Improvement labels:

| Label | Color | Owner | Used For |
|---|---|---|---|
| `product` | Orange | Jared | Specs, research, user stories |
| `architecture` | Green | Gilfoyle | Spikes, ADRs, code reviews |
| `implementation` | Blue | Dinesh, Big Head | Code, tests, features |
| `process` | Purple | Richard | Speed audits, bottleneck reports, compliance, process improvements |
| `needs-decision` | Red | Brandon | Blocked items requiring CEO decision |

### Issue Lifecycle

```
Backlog → Todo → In Progress → Done
```

- Issues move to `In Progress` **before** work starts — not after
- Issues move to `Done` only when the output is complete and reviewed
- `needs-decision` issues are resolved by Brandon in the chat, then re-labeled and moved to `Todo`

### Handoff Chain

Jared creates a spec → Gilfoyle creates a spike/ADR → Dinesh creates implementation issues. Each agent links their issue to the upstream issue. This creates a traceable chain from product decision to shipped code.

### Issue Naming

All issues use the format: `[Agent] Imperative description`

Examples:
- `[Jared] Competitor teardown — meeting intelligence tools`
- `[Gilfoyle] Spike — meeting bot options (recall.ai vs. custom)`
- `[Dinesh] Implement commitment extraction pipeline`
- `[Big Head] Implement transcript processing pipeline`
- `[Decision] Meeting bot approach: recall.ai vs. custom vs. other`

### Adding New Projects

When a new project is created under `projects/`:
1. Create a matching team in Linear
2. Create the four agent labels on that team
3. Note the Linear team name in the project's MEMORY.md under Decisions
